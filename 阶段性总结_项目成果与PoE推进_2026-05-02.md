# 阶段性总结：当前成果与 PoE 推进入口

更新时间：2026-05-02

用途：冻结当前可演示成果，给下一步 PoE 模块开发提供接续入口。本文档只记录已验证或当前代码中已经落地的内容；不把尚未验证的设想写成结论。

## 1. 当前总判断

当前系统已经可以按“原型仪器 + SDR 测长演示”的方式推进：

- 主体仪器框架已经形成：ESP32-S3 + TJC 串口屏，包含线序/通断、直流电阻、屏蔽类型、短路检测、长度页面。
- 长度模块已经达到演示目标：不追求严格计量，只展示“短线测短、长线测长”的趋势；长线结果可以作为论文和答辩演示的主要亮点。
- AD9959/AD8310 检波链路已经暂停，不再作为当前长度模块主线。
- 下一步应切到 PoE：先实现安全的 PoE 电压存在检测和供电模式识别，再考虑是否做分类等级展示。

## 3. ESP32-S3 + TJC 主体成果

主程序在 `ethernet_tester1/ethernet_tester1.ino`。当前页面状态机已经包含：

```text
PAGE_MAIN
PAGE_WIRE
PAGE_DCR
PAGE_TYPE
PAGE_SHORT
PAGE_LENGTH
```

已经落地的功能：

- TJC 串口屏通信：`Serial1`，115200 baud，GPIO18/GPIO17。
- 主页面与各功能页面切换。
- 线序/通断页面：8 芯 GPIO 扫描，识别 OK、OPEN、SHORT、MISWIRE 等状态。
- DCR 页面：四线对手动测量流程，当前使用 GPIO1 ADC 采样。
- 类型识别页面：通过 RJ45 外壳连续性判断 UTP / SHIELDED。
- 短路检测页面：GPIO 扫描线间短路，显示短路线对或 NO SHORT。
- 长度页面：通过 Wi-Fi TCP 或 UART 触发 SDR，显示 `LEN,xx.xx` 或 `ERR,...`。

主要硬件/引脚约束：

```text
Wire Map pins: GPIO4,5,6,8,9,7,10,11
TJC UART: RX GPIO18, TX GPIO17
SDR UART fallback: RX GPIO39, TX GPIO40
Type sense: GPIO12
DCR ADC: GPIO1
旧 AD8310 ADC: GPIO2
```

PoE 后续选 ADC 或 GPIO 时要避开这些已经占用的引脚，或者明确替换旧 AD8310 诊断用途。

## 4. 线序、短路、类型、DCR 当前状态

### 4.1 线序/通断

当前线序检测使用 8 芯逐线驱动/采样，目标是演示通断、开路、错线、短路状态。该部分属于主系统基础功能，可以作为答辩演示的稳定入口。

### 4.2 短路检测

短路页面独立于 Wire Map 页面，使用 GPIO 扫描判断 wire-to-wire short：

```text
无短路: TEST OK / NO SHORT
有短路: TEST FAIL / SHORT / 显示短路线对
```

### 4.3 类型识别

类型识别当前按 RJ45 shell continuity 实现：

```text
shell 连续: SHIELDED
shell 开路: UTP
```

`TYPE_USE_DEMO_JUMPER = false`，当前代码默认走真实 RJ45 外壳检测，不是杜邦线演示模式。

### 4.4 DCR

DCR 当前是四线对手动测量流程：

```text
Pair order: 1-2, 3-6, 4-5, 7-8
ADC pin: GPIO1
Rref: 46.4 ohm
Vsrc: 3.08 V
Offset: 0.25 ohm
Samples: 120
```

结论：DCR 可以作为演示功能，但不是高精度电阻计。论文中应写为“环路直流电阻估算/演示测量”。

## 5. 长度模块当前状态

### 5.1 当前方案

长度模块从 AD9959/AD8310 检波链路切换为 SDR/AD9361 2T2R 相位斜率法：

```text
DUT path: TX1A -> 被测线缆/夹具 -> RX1A
REF path: TX2A -> SMA 回环 -> RX2A
```

算法：

```text
发送多音 multitone
分别提取 DUT/REF 复数响应
计算 ratio = DUT / REF
拟合 phase(f) 的斜率
由群时延换算长度: length = tau * VF * c
```

当前参数在 `tmp_pluto_sdr_work/pluto_2x2_multitone_selftest.py`：

```text
LO_HZ = 100 MHz
SAMPLE_RATE = 30.72 MSPS
RF_BW = 28 MHz
N = 131072
N_AVG = 16
TX_GAIN_DB = -45
RX_GAIN_DB = 10
VF_CABLE = 0.66
```

### 5.2 演示优先的低频配置

为优先保证现场演示趋势，当前测长脚本已切到低频演示配置：

```text
TONE_START_HZ = 0.5 MHz
TONE_STOP_HZ = 6.25 MHz
TONE_COUNT = 32
MIN_VALID_TONES = 12
MIN_WINDOW_BW_HZ = 3 MHz
MAX_RESIDUAL_DEG = 3 deg
MAX_RESIDUAL_PEAK_DEG = 6 deg
```

全频严格拟合失败后，脚本还会进入低频演示回退：

```text
fit_mode = low_band_demo
Demo fallback = YES (low-band)
```

这表示该结果用于演示趋势，不应写成严格计量精度。

### 5.3 TCP 协议

ESP32 长度页面通过 TCP 触发 PC/SDR 侧测量：

```text
ESP32 -> server: START\n
server -> ESP32: LEN,7.95\n
server -> ESP32: ERR,LOW_SIGNAL\n
server -> ESP32: ERR,UNSTABLE_PHASE\n
```

启动命令：

```powershell
python sdr_length_tcp_server.py --host 0.0.0.0 --port 9000 --runs 1
```

模拟命令，用于不接 SDR 时验证 ESP32/TJC 页面：

```powershell
python sdr_length_tcp_server.py --host 0.0.0.0 --port 9000 --mock-length 8.00
```

### 5.4 已验证的长度结论

历史实验中已经得到以下可写入论文/答辩的结论：

```text
正常 2T2R 回环稳定，通道固定偏置很小。
交叉拓扑不可用于校准。
SMA 线差异约 2 mm 等效长度，可忽略。
系统零偏可取约 0.055 m。
1.12 m 短线在夹具稳定时可测到约 1.09 m 到 1.12 m。
约 7.91 m 长线 20 次测量平均约 8.147 m，误差约 +2.99%，重复跨度约 0.094 m。
```

当前更适合采用的表述：

```text
长度模块能够稳定表现短线与长线的测量趋势。
长线测量结果达到演示和论文可行性验证要求。
短线绝对精度受临时 RJ45/SMA 夹具影响明显。
```

不要写成：

```text
已经完成 10 m 到 50 m 全范围标定。
可以稳定分辨 1.10 m 与 1.12 m 的 2 cm 差异。
当前系统是完整计量仪器。
```

## 6. AD9959 / AD8310 状态

AD9959/AD8310 方案已暂停，原因如下：

- AD9959 输出曾被 PlutoSDR 验证有效。
- 两块 AD8310 模块在 Pluto RF 输入下，TPVout 没有随 TX ON/OFF 或 TX gain 明显变化。
- AD8310 供电、空载直流偏置基本正常，但没有证据证明模块已经产生有效 RF 检波响应。
- 当前打开的 `ethernet_tester2.ino/ethernet_tester2.ino.ino` 是 AD8310 TPin 诊断草图，不是主程序。

后续不建议继续把 AD8310 作为长度模块主线。若需要保留论文记录，可以把它写成“早期方案排查后转向 SDR 方案”。

## 8. 当前演示策略

建议答辩或现场演示顺序：

```text
1. 主页面进入 Wire Map，演示通断/线序。
2. 进入 Short Detect，演示无短路或人为短路。
3. 进入 Cable Type，演示 UTP / Shielded。
4. 进入 DCR，演示 1-2、3-6、4-5、7-8 四线对手动测量。
5. 进入 Cable Length，启动 SDR TCP server，演示短线与长线数值趋势。
6. PoE 完成后新增 PoE 页面或复用一个入口做 PoE 电压/模式显示。
```

长度模块当前目标只需要：

```text
短线显示约 1 m 量级。
长线显示明显大于短线，约 8 m 量级。
异常时显示 ERR，而不是输出明显错误长度。
```

## 9. PoE 下一步推进建议

### 9.1 PoE 目标先收窄

建议第一版 PoE 只做“安全检测 + 模式显示 + 电压显示”：

```text
NO_POE
MODE_A
MODE_B
MODE_A+B
UNKNOWN / UNSAFE
```

不要第一步就承诺完整 802.3af/at/bt 分类等级，因为真正分类涉及 PSE/PD 握手、电流签名和功率等级，不只是 ADC 读电压。

### 9.2 安全边界

PoE 电压不能直接进入 ESP32 GPIO/ADC。下一步硬件设计必须先确认：

```text
RJ45 到测量前端有高阻分压。
ESP32 ADC 输入被限制在 0 到 3.3 V。
前端能承受约 48 V 级 PoE 电压和插拔瞬态。
有串联限流、电阻功耗核算和钳位/保护设计。
测试时先用限流直流电源，不直接上 PoE 交换机。
```

### 9.3 建议检测对象

PoE 供电模式可先按 RJ45 线对电压判断：

```text
Mode A 候选: 1/2 与 3/6 之间的电压
Mode B 候选: 4/5 与 7/8 之间的电压
```

第一版显示内容建议：

```text
PoE: YES/NO
MODE: A/B/A+B/UNKNOWN
VA: xx.x V
VB: xx.x V
STATUS: OK / OVER_RANGE / REVERSE / NO_POE
```

### 9.4 软件接入点

主程序当前还没有 `PAGE_POE`。下一步代码应新增：

```text
PAGE_POE
poePageReset()
enterPoePage()
runPoeTest()
PoE 页面 TJC 对象常量
主循环中的 PoE 页面入口事件
```

如果 TJC 屏页面还未制作，可以先用串口输出和临时文本对象验证测量逻辑，再接 UI。

### 9.5 推荐开发顺序

```text
1. 画出 PoE 前端电路，确认分压比、限流、电阻功耗、ADC 最大电压。
2. 用限流直流电源模拟 0 V、12 V、24 V、48 V、57 V，验证 ADC 换算。
3. 写独立 PoE 诊断草图，只读 ADC 并输出 VA/VB。
4. 把诊断逻辑合入 ethernet_tester1 主程序。
5. 增加 PAGE_POE 和 TJC 显示。
6. 用 mock 电压/电源验证，再接真实 PoE injector 或交换机。
```

## 10. 下一步不要做的事

```text
不要继续把 AD8310 当作当前长度主线。
不要为了长度精度继续大改 SDR，当前演示目标已经够用。
不要把 PoE 电压直接接 ESP32。
不要在没有限流和分压保护前接真实 PoE 交换机。
不要把 Wi-Fi 密码、私人热点信息写进论文或公开文档。
```

## 11. 下一步最小可执行任务

推荐从 PoE 的硬件安全前端开始：

```text
任务名称: PoE voltage detect MVP
输入: RJ45 线对电压
输出: VA、VB、PoE mode、status
第一验收: 限流电源输入 48 V 时，ESP32 ADC 换算显示稳定电压，ADC 脚电压不超过 3.3 V。
第二验收: 主程序新增 PoE 页面，按 START 显示 NO_POE / MODE_A / MODE_B。
```

长度模块后续只建议做小维护：

```text
保留低频演示配置。
保留 TCP server。
保留 ERR 门限。
复测时只记录短线/长线趋势，不继续追求 2 cm 分辨率。
```
